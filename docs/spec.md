# LINE MAエージェント 企画仕様書

（旧仮称: サイトマイスター LINE配信機能）

**作成日**: 2026-07-06
**対象**: サイトマイスター周辺追加商材の企画検討（稟議前ドラフト）
**ステータス**: Draft v0.1

---

## 1. エグゼクティブサマリー

サイトマイスターの既存顧客（CRM君・MEOエージェント利用の店舗・中小企業）に対し、LINE公式アカウントを活用した**セグメント配信・シナリオ配信機能**を追加商材として提供する。

前回の議論の結論を踏まえ、方針は以下の通り：

- **機能としてはLINE拡張ツール型**（セグメント配信・シナリオ配信・タグ管理）を実装する
- **売り方は単体プロダクトではなく、CRM君の追加オプション**として位置付ける
- 差別化の核は「**CRM君に既にある顧客データを初日から使えるセグメント配信**」と、「**連絡する必要が無ければ、連絡しない**」という既存プロダクト哲学の一貫適用

この2点により、LINEマーケティングツール市場（後述の通り23社以上がひしめく典型的なレッドオーシャン）に単体で殴り込むのではなく、**既存顧客基盤へのクロスセルという別の勝ち筋**で戦う。

---

## 2. 背景・目的

- サイトマイスターはMEOエージェント・CRM君・RENOなど、店舗・中小企業向けの複数商材を展開中
- 既存顧客は既にLINE公式アカウントで顧客接点を持っているケースが多いが、標準機能では「属性・行動に応じた出し分け」「行動起点の自動配信」ができない
- この隙間を埋める形で追加商材を投入し、①既存顧客のLTV向上、②CRM君の解約防止（データ活用の幅が広がることでスイッチングコストが上がる）の2つを狙う

## 3. 市場環境・競合分析

### 3.1 市場の全体像

LINEマーケティングツール市場は**23社以上**が乱立する成熟市場。各社は概ね以下の軸で差別化している：

| 軸 | 内容 |
|---|---|
| 業界特化 | 飲食・美容室向けなど業種特化機能（予約連携、メニュー提案等） |
| セグメント精度 | 属性・行動・外部データを使った配信の細かさ |
| 自動化度 | シナリオ配信、AIによる配信タイミング最適化 |
| 料金モデル | 月額固定、成果報酬型（完全成果報酬もあり）、要問い合わせ型 |

### 3.2 主要競合比較表

| サービス | 提供元 | 初期費用 | 月額目安 | 強み | 導入規模 |
|---|---|---|---|---|---|
| Lステップ | 株式会社マネクル | 0円 | ¥5,000〜32,780 | 業界最大手、大量配信に強い、初心者向けUI | 有料43,000社以上 |
| L Message（エルメ） | 株式会社ミショナ | 0円 | 0〜33,000円 | オールインワン（集客〜決済）、無料枠あり | 100,000件以上 |
| MicoCloud | Micoworks | 要問合せ | 要問合せ | 大手金融機関実績、複数拠点一元管理、タグ分析 | 大手中心 |
| Liny | ソーシャルデータバンク | 要問合せ | 要問合せ | 属性自動収集、予約連携 | 3,500社以上 |
| TSUNAGARU | 株式会社デジタルシフト | 要問合せ | 要問合せ | ID×行動データのセグメント配信、専門チーム支援 | 大手企業中心（アダストリア等） |
| DECA for LINE | 株式会社ギブリー | 要問合せ | 要問合せ | 外部DB連携、ミニアプリ開発も対応 | 1,500社以上 |
| KAKERU | 株式会社ミライク | ¥220,000〜 | ¥11,000〜 | 機能を選んで組み合わせ、オリジナル開発対応 | 中小規模向け |
| Ligla | TimeTechnologies | 要問合せ | 要問合せ | AI/MLによる自動最適化、レコメンド配信 | AI訴求が強い |

### 3.3 競合の共通の弱点（＝サイトマイスターの機会）

1. **導入初期のデータがゼロから**：ほぼ全ツールがLINE上の友だち情報からデータを貯め始める設計。既存CRM/POSデータとの連携は「別途要問い合わせ」「要カスタム開発」が大半
2. **配信過多への誘導**：多くのツールが配信数アップを前面に出す。ユーザー体験（ブロック率上昇）への配慮を明示するツールは少ない
3. **中小企業には過剰スペック or 価格不透明**：大手向け（TSUNAGARU、MicoCloud）は要問い合わせで中小企業には敷居が高く、逆に安価なツール（Lステップ等）は業界特化・データ連携が弱い
4. **単体契約が前提**：既存業務システムとのシームレスな統合を謳うツールはほぼ皆無。「もう一つ管理画面が増える」ことへの負担感が導入障壁になっている

### 3.4 サイトマイスターの差別化ポジション

```
                 高機能・高価格
                      |
        TSUNAGARU  MicoCloud
                      |
  データ連携弱い ---- + ---- データ連携強い
                      |
        Lステップ   [サイトマイスターLINE]
        L Message     ← ここを狙う
                      |
                 低機能・低価格
```

**ポジショニング**：「もう一つ管理画面を増やさない、CRM君のデータがそのまま使えるLINE配信」

## 4. ターゲット顧客・ペルソナ

- **主要ターゲット**：既にCRM君・MEOエージェントを利用中の店舗・中小企業（新規のLINE市場開拓ではなくクロスセル）
- **ペルソナ例**：
  - 個人経営〜数店舗規模の飲食店・サロン・小売店オーナー
  - マーケティング専任者がおらず、店主やスタッフが片手間でLINE配信を運用
  - 「配信して良いお客様」と「そっとしておくべきお客様」の区別が曖昧なまま一斉配信し、ブロックが増えることに悩んでいる

## 5. プロダクトコンセプト

### コンセプトステートメント

> CRM君に蓄積された顧客データを使い、「連絡する必要がある人にだけ、必要なタイミングで」LINEを届ける。もう一つの管理画面は増やさない。

### プロダクト哲学の一貫性

既存のCRMエージェントの思想「連絡する必要が無ければ、連絡しない」をLINE配信にもそのまま適用する。これは競合の多くが配信量最大化を志向する中で、**明確な差別化かつ顧客体験上の優位性**になる。

具体的には：
- 配信前に「本当にこの人に送る必要があるか」をAIがスコアリングし、低スコアの顧客への配信は警告またはスキップを提案
- ブロック率の悪化を検知したら自動でセグメント条件を見直すよう促す（開封率はLINE Messaging APIで取得不可のため対象外）

## 6. 機能仕様

### 6.1 MVP（Phase 1）

| 機能 | 内容 | 優先度 |
|---|---|---|
| LINE公式アカウント連携 | Messaging API経由でLINE公式アカウントと接続（Webhook受信・プッシュ配信） | 必須 |
| CRM君タグ連携配信 | CRM君に登録済みのタグ・顧客属性をそのままセグメント条件として使える | 必須 |
| セグメント配信 | 属性・タグ・購買/来店履歴による絞り込み配信 | 必須 |
| シナリオ配信（簡易） | 登録後N日目、来店後N日目などのトリガー配信 | 必須 |
| 配信効果測定 | 送信成功率・ブロック率のダッシュボード表示（※LINE Messaging APIの制約上、開封率・クリック率は取得不可のため対象外） | 必須 |
| リッチメニュー管理 | CRM君管理画面から直接編集・出し分け | 推奨 |

### 6.2 Phase 2以降

| 機能 | 内容 |
|---|---|
| 配信要否スコアリング（AI） | 過去の反応データから「送るべきでない」顧客を自動検知 |
| チャットボット応答 | よくある質問への自動応答（Haiku等軽量モデル活用でコスト最適化） |
| 予約・会員証連携 | LINEミニアプリ（LIFF）との段階的統合。将来的にミニアプリ機能を追加商材として展開する布石 |
| 成果報酬型プラン | 小規模店舗向けに固定費を抑えた導入障壁の低いプラン |

### 6.3 あえて作らない機能

- 大規模企業向けの複数拠点一元管理（TSUNAGARU/MicoCloudの土俵では戦わない）
- LINE広告運用代行（広告代理店機能には踏み込まない）

## 7. CRM君との連携アーキテクチャ（案）

```
[LINE公式アカウント]
      ⇅ Messaging API (Webhook / Push)
[LINE配信サービス（Cloudflare Worker）]
      ⇅ Supabase (sfhtvtcmgueystyuhzvd 共有プロジェクト)
      　 - customers テーブル（CRM君と共有）
      　 - tags テーブル（CRM君と共有）
      　 - line_segments / line_scenarios（新規テーブル）
      　 - line_delivery_logs（配信ログ・効果測定用）
[CRM君 管理画面（crm.taskra.jp）]
      → 配信設定・セグメント作成・効果測定を同一UI内に埋め込み
```

- 既存のTaskra/Tavera/Flowraと同じ構成パターン（Cloudflare Worker + Supabase + GitHub Pages）を踏襲し、開発コストを抑える
- CRM君の顧客テーブルをそのまま参照するため、**顧客データの二重入力が発生しない**設計が最大の差別化ポイント
- secrets（LINE Channel Access Token等）は顧客ごとにSupabase側で管理し、マルチテナント対応が必須（既存顧客ごとに個別LINE公式アカウントを持つため）

| プラン | 想定顧客 | 月額目安 | 備考 |

## 9. マーケティング・GTM戦略

### 9.1 訴求メッセージ

- **既存顧客向け**：「今お使いのCRM君のデータが、そのままLINE配信に使えます。設定は追加で顧客リストを作り直す必要なし」
- **新規顧客向け**（CRM君とのセット提案）：「LINE配信を始めるなら、顧客管理も一緒に。バラバラのツールを一つに」

### 9.2 展開順序

1. 既存CRM君顧客への先行案内・ヒアリング（β版として無償提供し、フィードバック収集）
2. 効果事例（ブロック率低下等）をケーススタディ化
3. サイトマイスターの営業資料・提案書にLINE配信オプションを組み込み、新規商談でのアップセル材料にする

### 9.3 競合との対峙時のトークポイント

- 「Lステップ等を検討しているが、もう一つ管理画面を増やしたくない」という顧客の声を想定した比較トーク
- 「配信のしすぎでブロックが増える」という中小企業の実感ある悩みに対し、スコアリング機能で応える

## 10. 稟議上の論点

- **投資対効果**：新規開発コストは既存インフラ（Cloudflare Worker + Supabase）の流用で抑制可能。LINE Messaging APIの通数課金は顧客負担（LINE公式アカウント側の従量課金）のため、サイトマイスター側のコスト増は開発・保守工数が中心
- **リスク**：
  - LINE公式アカウントのマルチテナント管理（顧客ごとのChannel Access Token管理）の実装難易度
  - 既存競合（Lステップ等）を使っている既存顧客の乗り換えハードル
  - LINEヤフー社の規約変更リスク（Messaging APIの仕様変更等）
- **撤退ライン**：β提供後、CRM君既存顧客の一定割合（要目標値設定）が試用しない場合は本格投資を見送る
